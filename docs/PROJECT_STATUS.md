# Part 1 — Everything Done

## 1. Overview & Stack

Lumen is a **proof-of-concept** private-credit deal room. It models a full deal
lifecycle (sourcing → diligence → IC → docs → portfolio monitoring) for a
direct-lending shop, with the **Covenant Monitoring Suite** as the flagship,
deeply-built module. All data is **simulated and seeded deterministically** — no
live market connection, no real authentication.

| Layer | Technology |
| --- | --- |
| Framework | **Next.js 16** (`16.2.9`) App Router, React `19.2.4` |
| Language | **TypeScript** (`^5`), `zod` for input shapes |
| Styling | **Tailwind v4** (`@tailwindcss/postcss`), `tw-animate-css` |
| UI | Hand-written **shadcn-style** components over Radix primitives (`src/components/ui`) |
| ORM / DB | **Prisma 6** (`^6.19.3`) on **SQLite** |
| Charts | **Recharts** (`^3.8.1`) |
| Tables | `@tanstack/react-table` |
| Misc | `sonner` (toasts), `cmdk`, `lucide-react`, `next-themes`, `date-fns` |

Cross-cutting characteristics:

- **Simulated data.** Everything is generated by `prisma/seed.ts` with a
  deterministic hash-seeded RNG so demos and screenshots are reproducible.
- **Role-based access + information barriers.** A simulated role cookie drives
  `can()` permission checks and `canSeeDeal()` ethical-wall visibility
  (`src/lib/auth/roles.ts`).
- **Mocked-but-real-ready adapters.** A `BloombergClient` interface
  (`src/lib/bloomberg/`) and a `CopilotService` interface (`src/lib/copilot/`)
  are implemented by mocks today, but are architected so a real BLPAPI / Data
  License client and a Claude-backed copilot can be dropped in behind the same
  contract.
- **Vercel-deployable.** `vercel.json` plus a `/tmp` SQLite bootstrap in
  `src/lib/db.ts` make the demo run on Vercel's read-only serverless filesystem
  (writes are ephemeral per instance — see backlog).

## 3. The Covenant Monitoring Suite

The focus of recent work. It independently **recomputes** each covenant from
Bloomberg-style fundamental fields via a safe formula engine, **reconciles** the
recompute against the borrower-reported figure, classifies status with a strict
precedence, and **forecasts** headroom under scenarios — surfacing the kinds of
discrepancies a spreadsheet tracker would silently miss.

### 3.1 The pure isomorphic engine (`src/lib/covenants/`)

**Isomorphism rule:** nothing under `src/lib/covenants/` imports Prisma,
`server-only`, or any adapter. The barrel (`index.ts`) is safe to import from
**both** server queries and client components (the scenario slider runs the same
`evaluate`/`projectCovenant` in the browser). The comment at the top of
`index.ts` and `types.ts` enforces this contract.

| File | Responsibility |
| --- | --- |
| `types.ts` | Shared types: `Operator`, `CovenantCategory`, the 9-value `CovenantStatus` union, `Ast`, `ParsedDefinition`, `EvalResult`, `ThresholdStep`, `SpringingCondition`. |
| `fields.ts` | The **field library** — Bloomberg-style mnemonics (`EBITDA`, `EBITDA_ADJ`, `TOT_DEBT`, `CASH`, `NET_DEBT` (derived `TOT_DEBT - CASH`), `INT_EXP`, `FIXED_CHARGES`, etc.) with units, categories, and the real `bbg:` field they map to. Also `RATIO_TEMPLATES` (one-click formula presets) and `EBITDA_FIELDS`. |
| `parser.ts` | **Safe formula parser**: tokenizer → shunting-yard → AST. **Never uses `eval`/`Function`.** Only `+ - * /`, parens, numbers, and identifiers that resolve against the field library are accepted; anything else throws `FormulaError`. Unary minus is encoded as `(0 - x)`. Includes `collectFields()` and `validateFormula()` for live UI feedback. |
| `evaluate.ts` | `evalAst()` (recursively resolves derived fields, divide-by-zero → `NaN`), `headroom()`, `passes()` (with a `1e-9` epsilon), `springingActive()`, **`solveBreakeven()`** (bounded bisection over the EBITDA field, 60 iterations, formula-agnostic, returns `null` if no sign change), and the top-level `evaluate()` returning an `EvalResult`. |
| `schedule.ts` | `resolveThreshold()` — **step-down schedule resolution** keyed on `periodEnd` (the tested period, never the test date). The latest effective step at-or-before `periodEnd` wins; pre-schedule periods use the opening level. |
| `reconcile.ts` | `reconcile(recomputed, reported, tolPct=2, tolAbs=0.1)` → `{ recomputed, reported, delta, deltaPct, flag }`. The `flag` fires when the gap exceeds tolerance. |
| `status.ts` | **`deriveCovenantStatus()`** — single source of truth for status, with strict **precedence** (below). `NEAR_BREACH_BAND_PCT = 10`. |
| `forecast.ts` | `applyScenario()` (EBITDA haircut + rate shock, pure/immutable) and `projectCovenant()` (project across periods vs the step-down schedule, marking `trips`). |
| `json.ts` | Typed, guarded parse/serialize for SQLite's stringified-JSON columns (`parseAst`, `parseSchedule`, `parseSpringing`, `parseBasket`, `parseSnapshot`, `toJson`). Every parse is `try/catch`-guarded so a malformed row never 500s a page. |
| `index.ts` | Barrel + `toParsedDefinition()` (DB row → `ParsedDefinition`, resilient: re-parses formula text if the stored AST is missing/corrupt, falls back to a safe constant AST). |

**`deriveCovenantStatus` precedence** (in `status.ts`):

```
Waived
  → (Reporting) Late / Missing / Pass(Delivered) / Upcoming
  → springing-not-met  ⇒ N/A-springing
  → no actual / null    ⇒ Upcoming
  → Breach              (ALWAYS judged on the recomputed/independent value)
  → Recon-flag          (passes, but reconciliation gap exceeds tolerance)
  → Near-breach         (headroom < 10%)
  → Pass
```

Breach is always evaluated against the **recomputed** value; the reported value
and delta are surfaced separately.

### 3.2 The Bloomberg `getFundamentals` anchored mock

`MockBloombergClient.getFundamentals()` (`src/lib/bloomberg/client.ts`)
**back-solves a coherent field set from the reported financials** (the
`anchor`: ebitda, netLeverage, interestCoverage, liquidity, capex, revenue)
rather than inventing numbers. It:

1. Reads the anchor (or synthesizes plausibly when absent).
2. Derives `netDebt = netLeverage × ebitda`, `cash = liquidity × 0.55`, and
   back-solves `TOT_DEBT`, `INT_EXP = ebitda / coverage`, `SCHED_AMORT`,
   `FIXED_CHARGES`, `SR_DEBT`, revolver fields, etc.
3. Applies a **±1% deterministic jitter to a single field (total debt)** so the
   independent recompute lands ~0.5–1.5% off the reported figure — i.e. it
   produces a small, realistic **recon delta** rather than diverging wildly.

The RNG is seeded per `borrower:periodEnd`, so the same inputs always return the
same fields. The action `recordFundamentals()` (`covenant-ops.ts`) walks every
actual `FinancialStatement`, calls `getFundamentals` with the anchor, and
persists each field as a `FundamentalFact(source="BBG")`.

### 3.3 The four trigger types

The `CovenantCategory` union models four trigger types; each is handled
distinctly by the engine and queries:

| Type | Category | How it's modeled |
| --- | --- | --- |
| **Maintenance** | `Maintenance` | Tested every period against a (possibly stepped-down) threshold. Recomputed → reconciled → status. e.g. Total Net Leverage, Fixed Charge Coverage. |
| **Springing** | `Springing` | Only tested when a `springingCondition` (`{ field, op, value }`, e.g. `RCF_UTIL_PCT > 40`) is met. When not met, `springingActive=false` ⇒ status `N/A-springing`. |
| **Incurrence** | `Incurrence` | A test gating an action (e.g. debt incurrence ratio); carries an optional `basketConfig` (`{ type, capFormula, builderBasis, capacity }`). Seeded with a single latest-period test. |
| **Reporting / Information** | `Reporting` | Not a ratio — modeled via a `ReportingObligation` (kind, `dueDaysAfter`, FY-end month) with per-period `ReportingDelivery` rows. Status derives from delivery: `Delivered`→Pass, `Late`, `Missing`, else `Upcoming`. |

### 3.4 "Win on the margin" features

These are the differentiators over a basic covenant tracker:

- **Reconciliation (recomputed vs reported + recon-flag).** Every maintenance/
  springing test stores `recomputedValue` (engine), `reportedValue` (cert), and
  `reconDelta`. When the delta exceeds tolerance the status becomes
  `Recon-flag` — even though the test "passes" — catching a borrower figure
  under review. See `reconcile.ts`, `evalPeriod()` in `queries/covenants.ts`,
  and the `reconciliation-table.tsx` UI.
- **EBITDA add-back bridge + cap enforcement + aggressive/uncapped flag.**
  `EbitdaAdjustment` rows build GAAP → Adjusted EBITDA. `addback-waterfall.tsx`
  renders the bridge, enforces the **add-back cap** (capped items ≤ 25% of GAAP
  EBITDA, configurable via `capPctOfGaap`), and surfaces a quality-of-earnings
  warning for **uncapped / aggressive run-rate** items (e.g. pro-forma M&A).
- **Forecasting.** `forecast/page.tsx` + `scenario-slider.tsx`:
  - **Break-even gauges** — `solveBreakeven()` shows the EBITDA at which each
    covenant trips, holding other inputs fixed.
  - **Projected headroom** — `projectCovenant()` extends across actual +
    projected periods vs the step-down schedule.
  - **Scenario slider / trip-matrix** — client-side EBITDA-haircut (0–40%) and
    rate-shock (bps) sliders re-run the *same* engine in the browser and mark
    which covenant/period combinations `trip`.
- **AI extraction (mocked).** `runCovenantExtraction()` action calls the mocked
  copilot for `extract_covenant_terms` (returns formula-ready definition drafts
  incl. schedules + springing conditions) and `extract_cert_figures` (parses a
  compliance cert into reported values + add-backs). Output is presented for
  analyst review, not auto-applied.

### 3.5 Reporting outputs

- **Per-deal Compliance Pack (print).** `covenants/pack/page.tsx` +
  `print-button.tsx`. `getCompliancePack()` attaches each test's persisted
  `formulaSnapshot` (`{ formula, inputs, value, thresholdApplied }`) so the pack
  shows the actual math behind every number — a print-ready compliance artifact.
- **Portfolio board heatmap.** `getCovenantBoard()` + `board-heatmap.tsx` on
  `/covenants`. One row per deal, worst-status-first (via a `STATUS_SEVERITY`
  map), with per-covenant chips, min headroom, recon-flag count, and
  reporting-late count, plus aggregate stats.

### 3.6 Prisma models & the legacy bridge

The suite added these models to `prisma/schema.prisma` (the "Covenant Monitoring
Suite" section), **coexisting with the legacy `Covenant` / `CovenantTest`**:

| Model | Purpose |
| --- | --- |
| `CovenantDefinition` | Formula-driven definition: `formula`, `formulaAst` (JSON), `fieldRefs`, `ebitdaBasis`, `operator`, `unit`, `threshold`, `thresholdSchedule` (JSON), `springingCondition` (JSON), `basketConfig` (JSON), `source`. **`legacyCovenantId`** bridges to the old `Covenant` row. |
| `CovenantDefTest` | Persisted per-period result: `recomputedValue`, `reportedValue`, `reconDelta`, `thresholdApplied`, `formulaSnapshot` (JSON), `headroomPct`, `status`. `@@unique([definitionId, periodEnd])`. |
| `FundamentalFact` | Per-borrower/period/field value with `source` (`BBG`/`Cert`/`Override`/`Computed`) and `isOverride`. Overrides win on read. |
| `EbitdaAdjustment` | Add-back ledger row: `amount`, `category`, `capped`, `aggressiveFlag`, `uncapped`, `source`. |
| `ReportingObligation` | `kind`, `dueDaysAfter`, `fiscalYearEndMonth`; one-to-one with a definition. |

**The bridge.** The legacy `Covenant`/`CovenantTest` models still power the

SQLite has no JSON type, so JSON-shaped columns are stored as `String` and
parsed via the guarded helpers in `json.ts`.

### 3.8 Seeded "drama" on Project Atlas

`prisma/seed.ts` seeds the flagship deal **Project Atlas / Meridian Health
Partners** with deliberate edge cases so every feature has something to show:

- **Recon-flag quarter** — `JITTER_PERIOD = "Q2 2025"` applies a `1.04×` jitter
  to `TOT_DEBT`, so the recomputed leverage exceeds the reported (clean) figure
  beyond tolerance → a `Recon-flag` on Total Net Leverage that period.
- **Near-breaches vs the step-down** — the leverage step-down schedule
  (`5.75 → 5.25 → 4.5`, effective `2023-07-15 / 2024-12-31 / 2025-09-30`) bites
  as the borrower delevers, producing tight headroom / `Near-breach` statuses.
- **Springing activation** — `SPRINGING_PERIOD = "Q3 2025"` sets
  `RCF_UTIL_PCT = 45` (> 40), activating the "Springing Interest Coverage" test
  for that quarter (otherwise `N/A-springing`).
- **Late filing** — the `Compliance Cert` reporting obligation has Q2 2025
  delivered 7 days after the due date → status `Late`; a Q1 2026 delivery is
  `Pending`.
- **Aggressive uncapped add-back** — the add-back ledger includes "Pro forma M&A
  EBITDA (run-rate, unrealized)" (≈11% of adjusted) flagged `aggressiveFlag` +
  `uncapped`, driving the quality-of-earnings warning in the waterfall.

Seeded covenants: Total Net Leverage (Maintenance, stepped), Fixed Charge
Coverage (Maintenance), Springing Interest Coverage (Springing), Incurrence —
Debt Basket (Incurrence, with basket config), plus three Reporting obligations
(Quarterly FS, Annual Audited, Compliance Cert). All `CovenantDefTest` rows are
generated **through the real engine** at seed time (`evaluate` + `reconcile` +
`deriveCovenantStatus`), so seeded statuses match runtime recomputation.

## 4. Architecture & Conventions

**The server-query → server-component → client-island pattern.**
Server components (the `page.tsx` files) call typed functions in
`src/server/queries/*` and pass plain serializable data to small `"use client"`
islands (e.g. `scenario-slider.tsx`, `suite-actions.tsx`) for interactivity.

**Mutations are server actions** in `src/server/actions/*`, each wrapped with the
shared helpers (`src/server/actions/helpers.ts`):

- **`guard(action, resource)`** — resolves the active role, enforces `can(...)`,
  throws `Forbidden` if denied, and returns the actor's display name.
- **`logActivity(dealId, actor, role, action, target)`** — writes an
  `ActivityLog` row for the audit trail.
- **`revalidatePath(...)`** — covenant actions call `bust(dealId)` to invalidate
  every covenant sub-route plus `/covenants`.

**Access control** (`src/lib/auth/`):
- `can(role, action, resource)` — a `Role → Resource → Action[]` permission
  matrix over 5 roles (Deal Lead, Analyst, IC Member, Compliance, Read-only).
- `canSeeDeal(role, deal)` — the **information barrier**: a `isPrivileged`
  (wall-crossed) deal is hidden from `Read-only`. Applied in covenant queries
  (`getCovenantSuite`, `getCovenantBoard`, `getForecastInputs`).
- The active role comes from a **cookie** (`getActiveRole()` in
  `auth/server.ts`); the client context is `auth/context.tsx`.

**Deterministic seeds.** The Bloomberg mock and the seed both use a stable
hash-seeded RNG so output is reproducible.

**Engine isomorphism rule.** Restated: `src/lib/covenants/*` is pure — no Prisma,
no `server-only`, no adapters — so it runs identically on server and client.

**Covenant-related directory map:**

## Known issues (from the recent code review)

### [MED] Board can show stale status vs the live suite

`getCovenantSuite()` recomputes **live** from current `FundamentalFact`s +
overrides on every read, whereas `getCovenantBoard()` reads the **persisted**
`CovenantDefTest.status` for speed (its comment: *"Use persisted def-tests for
speed (no per-deal live evaluation)"*). If fundamentals or overrides change
**without** re-running reconciliation, the board can show a stale status while
the per-deal suite shows the current one.
**Options:** have the board recompute live, or add a "stale" indicator (e.g.
compare a fundamentals/override timestamp against the test's last reconciliation).
*Refs:* `getCovenantSuite` / `getCovenantBoard` in `src/server/queries/covenants.ts`.

### [LOW] `addback-waterfall.tsx` divide-by-zero / NaN when GAAP EBITDA is 0

`AddbackWaterfall` computes `(totalAddbacks / gaapEbitda)`,
`(uncappedTotal / adjEbitda)`, and a `% of GAAP` figure with **no guard** for
`gaapEbitda === 0` (or `adjEbitda === 0`), which would render `NaN%` / `Infinity`.
Not reachable with the current seed (GAAP EBITDA is always positive), but worth
guarding before real data.
*Ref:* `src/components/covenants/addback-waterfall.tsx`.

### [LOW] `runReconciliationInternal` ignores Waived tests

In `runReconciliationInternal` the loop does
`if (t.status === "Upcoming" || t.status === "Waived") continue;`, so **Waived
tests are skipped** and `waived` is **never passed** to `deriveCovenantStatus`
during the persisted recompute. A waived test won't refresh its recompute/delta,
and if a covenant should re-derive `Waived` precedence on persisted recompute it
won't. (The live `getCovenantSuite` path *does* pass `waived` correctly.)
*Ref:* `runReconciliationInternal` in `src/server/actions/covenant-ops.ts`.

### [LOW] `getForecastInputs` projected-period synthesis is approximate

For **projected** periods (no facts yet), `getForecastInputs` synthesizes facts
by scaling **only** the EBITDA family (`EBITDA`, `EBITDA_ADJ`, `SALES_REV_TURN`)
by EBITDA growth and easing `TOT_DEBT` (by `(scale-1)*0.3`). `INT_EXP`,
`FIXED_CHARGES`, and `CASH` are **carried flat**, so projected coverage and
leverage ratios in the forecast are approximate (interest doesn't move with rates
or balance in the projection).
*Ref:* `getForecastInputs` in `src/server/queries/covenants.ts`.

## Larger roadmap items (not yet built)

Verified absent from the codebase (greps returned nothing for equity-cure,
overcure, test runner, Anthropic SDK, upload action):

- **Equity-cure mechanics & breach lifecycle.** No cure clock, no overcure caps,
  no breach state machine. *Explicitly de-scoped earlier.* Would build on the
  existing `Breach` status + `LifecycleEvent` (Waiver/Amendment).
- **Override-fundamental UI.** The `overrideFundamental` server action exists
  (`covenant-ops.ts`) and the model supports `isOverride` (overrides win on
  read), but there is **no UI button/form** to invoke it — only the action.
- **LP/BDC structured export + exception report.** Only the per-deal
  **Compliance Pack** (print) and the **board heatmap** were built. No
  structured (CSV/XLSX/PDF) LP- or BDC-style export, and no standalone exception
  report.
- **Wire AI extraction / copilot to the live Claude API.** `CopilotService` is
  fully mocked (`MockCopilotService`) and `/api/copilot` is a mock endpoint.
  No `@anthropic-ai/sdk` dependency. The interface is designed for a
  `ClaudeCopilotService` (streaming) to drop in behind the same contract.
- **Real document upload / parsing.** Documents are seeded with synthetic
  `bodyText`; there is no upload action and no real file/PDF parsing pipeline.
- **Real authentication.** Auth is a **simulated role cookie**
  (`auth/server.ts` + `roles.ts`); there is no identity provider, session, or
  per-user accounts.
- **SQLite → Postgres migration.** Writes on Vercel are ephemeral per serverless
  instance (`/tmp` bootstrap in `db.ts`). Durable write-back requires migrating
  the datasource to Postgres.
- **Automated tests.** No test runner is configured (no `vitest`/`jest`, no
  `test` script). *Note: a separate effort is adding integration tests* — the
  pure engine in `src/lib/covenants/` is the natural first target given its
  isomorphism and lack of I/O.
